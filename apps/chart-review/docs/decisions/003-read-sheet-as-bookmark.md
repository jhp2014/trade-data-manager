# ADR-003: 북마크는 별도 기능이 아니라 Read Sheet로 해결

## 상태

Accepted (2026-06-03)

---

## 맥락

복기 중 "관심 종목 묶음을 따로 모아 빠르게 다시 훑고 싶다"는 북마크 요구가 있었다. 처음에는 DB 기반의 협업 북마크 기능을 구상했다:

- `[최근][북마크그룹]` 소스 탭을 가진 통합 스위처
- `a`/`d`로 소스 전환 + 활성 소스 유지
- `f` 키로 현재 차트를 그룹에 추가/제거(토글)
- 버튼 + 인라인 입력으로 그룹 관리
- 나중에 북마크 그룹을 Sheet로 내보내기

이 방향으로 **DB 테이블 2개(`review_bookmark_group`/`review_bookmark_item`) + 라우트 + 스토어 + UI까지 전부 구현**했다(빌드·마이그레이션 통과).

---

## 검토한 대안

**A. DB 기반 북마크 기능 (구현했다가 폐기)**
- 새 테이블·라우트·스토어·통합 스위처·`f` 토글·그룹 관리 UI 일체.
- 폐기: [ADR-002](./002-sheet-to-db-source-of-truth.md)로 이미 **Read Sheet가 "어떤 그룹을 볼지"를 정하는 작업셋 메커니즘**을 갖췄다는 걸 깨달았다. 북마크는 그것의 특수 사례에 불과했다.

**B. Read Sheet를 북마크로 사용 — 채택**
- 보고 싶은 `(stockCode, tradeDate)` 행만 담은 시트(또는 탭)를 연결하면 그게 곧 북마크 컬렉션이다.
- **새 코드 0줄**. 기존 작업셋 로딩(`resolveWorkingSetKeys`), `a`/`d` 순회, GroupId 붙여넣기, Tab 히스토리가 그대로 북마크 탐색이 된다.
- 컬렉션 여러 개 = 같은 스프레드시트의 **탭 여러 개**. 시트는 사람이 편집·공유하기도 쉽다.

---

## 결정

별도 북마크 기능을 두지 않는다. **Read Sheet(작업셋) = 북마크**로 운용한다.

- DB 북마크 구현 전체를 되돌렸다(스키마/리포지토리/라우트/스토어/컴포넌트/마이그레이션 파일).
- 이미 라이브 DB에 적용됐던 빈 테이블 `review_bookmark_group`/`review_bookmark_item`은 `DROP`했다.

---

## 결과

**장점**
- 추가 코드·테이블 없이 북마크 요구를 충족. 유지보수 표면이 늘지 않았다.
- 쿠키 기반이라 사람마다 다른 북마크(작업셋)를 동시에 쓸 수 있고, 시트라서 협업 편집·내보내기도 자연스럽다.

**한계 / 트레이드오프**
- 북마크 단위가 계정이 아니라 **브라우저 쿠키**다(별도 인증 없음). 각자 자기 쿠키를 설정해야 독립 작업셋이 된다.
- `f` 토글 같은 "현재 차트를 즉석에서 북마크에 추가"는 없다 → 시트에 행을 추가하는 방식으로 대체.
- 북마크 = 작업셋이므로, 북마크를 바꾸면 보이는 복기 대상 전체가 바뀐다(부분 오버레이가 아님).
